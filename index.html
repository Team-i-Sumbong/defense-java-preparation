Module 1

Architecture Mindset: Organize by Responsibility

Principle

Software becomes easier to explain when each layer has one main job.

Students should first understand that good architecture separates responsibilities. The UI handles interaction, the service layer applies rules, the repository handles storage, and the model represents data.

Why It Matters

  • It prevents one file from doing everything.
  • It makes bugs easier to locate.
  • It allows each part of the program to stay focused.
  • It gives a clean explanation during defense.

i-Sumbong Case

App.java -> AuthPortalView -> AuthService -> Repository -> JSON data

The project already shows layered thinking. App.java starts the app, AuthPortalView handles screen interaction, and AuthService decides whether login or registration is allowed.

Defense Answer

We separated the system into layers so each part has a clear responsibility. This improves maintainability and makes the program easier to debug, extend, and explain.

Common weak answer: "We have UI, service, and repository because that is how the code is written."
Better answer: "We used layers because each layer solves a different problem: interaction, business rules, and storage."

Module 2

Classes and Models: Represent Real Things Clearly

Principle

A class is a blueprint, and a model class represents a real entity in the domain.

Students should learn to ask, "What real thing does this class stand for?" If the answer is "a report," "a user," or "a status log," then the class is modeling part of the system domain.

i-Sumbong Case

Class Represents
User A person using the system
IssueReport A submitted concern
StatusLog A record of status changes

Concept Check

What is the difference between a class and an object?

Click to reveal

A class is the design. An object is one actual instance created from that design.

Concept Check

Why is IssueReport a model?

Click to reveal

Because it stores the data of a real report, such as title, category, location, and status.

Defense Answer

We used model classes to represent real system entities in a structured way. This keeps the data consistent and allows other layers to work with clear objects instead of random variables.

Module 3

Method Responsibility: One Method Should Have a Clear Purpose

Principle

Read a method by asking what it receives, what it decides, and what it returns.

  1. Read the method name.
  2. Read the parameters.
  3. Read the return type.
  4. Translate the method into plain English.

i-Sumbong Case

public boolean registerResident(User user)

In plain English: receive a user object, validate it, assign defaults and IDs, save it, and report whether registration succeeded.

Strong design signal

AuthService contains rule-based methods like login and registration, which means business logic is not trapped in the UI.

Professor's critique

AuthPortalView is effective, but long. This is a good discussion point about when UI classes become harder to maintain.

Module 4

Encapsulation: Control Access to Data

Principle

public String getReportId() {
    return reportId;
}

public void setReportId(String reportId) {
    this.reportId = reportId;
}

Why It Matters

  • Encapsulation keeps object data organized.
  • It avoids uncontrolled direct changes from everywhere.
  • It makes future validation easier to add.
  • It supports cleaner object-oriented design.
Defense Answer: We used getters and setters to support encapsulation, meaning object state is accessed through controlled methods instead of being modified directly throughout the whole system.

Module 6

Design Critique: Read the Code Like an Engineer

Questions To Ask

  • Does this class have a focused responsibility?
  • Is business logic mixed too much with UI handling?
  • Does the method name match what it really does?
  • Is data storage separated from interaction logic?

Real Critique of the Project

  • AuthService is a strong example of keeping rules outside the UI.
  • ReportRepository is easy to explain because it focuses on persistence.
  • AuthPortalView handles many screen concerns, which makes it a good example of a class that may later need refactoring.

1-Minute Sample

i-Sumbong is a Java-based barangay reporting system that follows separation of concerns. The app starts in App.java, the UI layer gathers input, service classes apply rules such as login and verification, repository classes handle JSON persistence, and model classes represent entities like users and reports. This design keeps the system easier to maintain and explain.
