Module 1

Architecture Mindset

Key Idea

Good architecture separates the program by responsibility.

The important point is not the number of files. The important point is that each layer has one main job.

Project Example

  • AuthPortalView handles interaction
  • AuthService applies rules
  • ReportRepository handles persistence
  • IssueReport represents data

Defense Line

We separated responsibilities so each part stays focused, easier to maintain, and easier to explain during defense.

Open support note

Weak answer: We used layers because that is how Java projects look.

Better answer: We used layers because interaction, rules, storage, and data representation should not compete inside one class.

Module 2

Classes and Models

Key Idea

A class should represent one clear role, and a model should represent a real thing in the system.

Ask this first: what responsibility or entity does this class stand for?

Project Example

Class What It Represents
User A person using the system
IssueReport A submitted concern
StatusLog A record of report updates

Defense Line

We used model classes to keep important data in structured objects, which makes the system more consistent and easier for other layers to use correctly.

Open support note

Class vs object: A class is the blueprint. An object is one actual instance created from that blueprint.

Fast example: IssueReport is the class, while one submitted complaint is an object.

Module 3

Methods and Encapsulation

Key Idea

A strong method has a clear job, and encapsulation keeps object data controlled.

Read methods by checking what they receive, what they decide, and what they return.

Project Example

public boolean registerResident(User user)

This method receives a user, validates the data, prepares default values, saves the record, and returns whether the action succeeded.

Defense Line

We kept behavior inside focused methods and controlled data through getters and setters so the code stays organized and ready for future validation rules.

Open support note

Why encapsulation matters: It prevents object state from being changed carelessly across the whole program.

Simple phrasing: Access happens through controlled methods instead of direct field changes everywhere.

Module 4

System Flow

Key Idea

1. UI receives input

The user enters information into the screen, and the UI passes that input to the correct service.

Open support note

Login pattern: UI input -> service validation -> repository lookup -> result in UI.

Report pattern: UI input -> service validation and defaults -> repository save -> success feedback.
